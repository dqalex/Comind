# TeamClaw — 让 AI Agent 从"聊天助手"升级为"团队成员"

> **一句话定位**：TeamClaw 是一个 AI Agent 协作管理平台，让你像管理团队成员一样管理 AI，用看板追踪进度、用知识库共享文档、用对话信道下达指令——而不是在聊天框里爬楼。

---

## 为什么做 TeamClaw？

### 从一个真实痛点出发

AI IDE（如 Cursor、Windsurf、CodeBuddy）已经证明了 AI 在编程场景中的巨大价值。但 AI 的能力不止于写代码——**OpenClaw 这样的 Agent 平台让 AI 可以像一个独立的人一样，端到端地完成任务**：调研、写文档、执行部署、定期巡检……

问题来了：

> **当你有多个项目、多个 AI Agent 同时在干活时，你怎么管？**

如果只有一个项目，通过聊天框与 AI 沟通勉强够用。但项目多了之后：

- 📜 **信息爆炸**：几个 Agent 同时在聊天框里汇报，你需要不断爬楼才能找到关键信息
- 📊 **进度失控**：哪个任务做完了？哪个卡住了？全靠记忆或者等 AI 发日报
- 📄 **文档散落**：AI 产出的文档通过聊天框发送，找起来极不方便
- 🔄 **主动性不足**：AI Agent 经常"做完就停了"，需要反复催促

**这就是 TeamClaw 要解决的问题——给 AI Agent 一个正经的工作台。**

### 市场调研的发现

我们调研了市面上的任务管理工具（Jira、Linear、Notion、飞书项目等），发现一个共同的问题：

> **它们对 AI 的定位都是"聊天机器人"，而不是"团队成员"。**

有些工具支持了 MCP 接口，但业务流程对 OpenClaw 这类自主型 Agent 并不友好——Agent 需要的不是被问一句答一句，而是**被分配任务、自主推进、汇报进度、提交成果**。

TeamClaw 的设计理念从一开始就不同：**AI 是团队的一等公民。**

## TeamClaw 能做什么？

### 🎯 核心场景

| 场景 | 传统方式 | TeamClaw 方式 |
|------|---------|------------|
| 分配任务给 AI | 在聊天框里打一大段文字 | 在看板上创建任务卡片，指定 AI 成员 |
| 追踪 AI 进度 | 爬聊天记录，或等 AI 发日报 | 看板实时显示：待办 → 进行中 → 审核中 → 完成 |
| AI 产出文档 | 聊天框里发一堆 Markdown | AI 直接写入知识库 Wiki，支持双向链接和图谱 |
| 多项目管理 | 每个项目开一个聊天窗口 | 统一看板，按项目筛选，一目了然 |
| 催促 AI 干活 | 手动发消息"进度怎么样了？" | 心跳机制 + 定时任务自动催促 |
| AI 交付审核 | 在聊天里 review AI 的输出 | 正式的交付审核流程：待审 → 通过/拒绝/修改 |

### 🏗️ 10 大功能模块

1. **任务看板** — 四列看板 + 拖拽排序 + 检查项 + 评论 + 操作日志
2. **知识库 Wiki** — Markdown 编辑器 + 双向链接 + 知识图谱 + 全文搜索
3. **智能对话** — 多模式对话（聊天/API操作/混合）+ 实体绑定 + Thinking 级别控制
4. **定时任务** — 可视化 Cron 调度 + 心跳唤醒 + 执行历史
5. **成员管理** — 人类 + AI 统一管理 + Agent 实时状态监控
6. **项目管理** — 多项目切换 + 跨项目视图
7. **文档交付** — 审核流（待审/通过/拒绝/修改）+ 版本追踪
8. **会话管理** — 活跃会话列表 + 参数在线调整（Thinking/Verbose/Reasoning）
9. **技能市场** — 技能列表 + 依赖安装 + API Key 管理
10. **仪表盘** — 连接管理 + 统计卡片 + 快捷入口 + 系统状态
11. **Skill 管理** — Skill 注册/审批/信任管理/快照监控（v3.0）
12. **审批系统** — 通用审批流程，支持多种审批场景（v3.0）

### 🔑 关键能力

**MCP 指令系统（37 个工具）**
AI Agent 不是通过模拟点击来操作 TeamClaw，而是通过标准化的 MCP 工具直接调用——创建任务、更新进度、写文档、提交交付，一切都是 API 级别的精确操作。支持内部调用和外部 API（Bearer Token 认证）两种方式。

**SOP 工作流引擎**
标准化操作流程引擎让 AI Agent 按预定义的多阶段工作流自动执行复杂任务：
- 7 种阶段类型：input / ai_auto / ai_with_confirm / manual / render / export / review
- Know-how 知识库分层（L1-L5）
- 内置模板：竞品调研、内容营销、周报月报、Bug 分析、数据分析

**实时状态同步**
- WebSocket 实时连接 OpenClaw Gateway（Protocol v3）
- SSE 推送本地数据变更
- 标签页切换自动刷新
- AI Agent 状态（空闲/工作中/等待中/离线）一目了然

**双数据源架构**
- 本地 SQLite：任务、项目、文档、聊天等持久化数据
- OpenClaw Gateway：Agent 状态、定时任务、会话、技能等实时数据
- 两套数据体系无缝融合

**多用户权限系统（v3.0 新增）**
- 三种角色：admin（管理员）/ member（成员）/ viewer（只读）
- 初始化向导：首次部署自动引导创建管理员
- 安全码二次验证：敏感操作需要输入安全码

**Skill 管理系统（v3.0 新增）**
- Skill 注册与验证：自动验证 SKILL.md 结构规范
- 审批流程：Skill 发布需管理员审批
- 信任管理：信任/拒绝未知来源 Skill
- 快照监控：定期检测 Agent 已安装 Skill 变更，发现风险告警

**通用审批系统（v3.0 新增）**
- 4 种审批类型：Skill 发布、Skill 安装、项目加入、敏感操作
- 灵活策略：审批人规则、超时设置、自动处理
- 完整审计：审批历史全记录

## 解决了什么问题？

### 对于个人用户

> "我有 2-3 个 AI Agent 在帮我做不同的事，TeamClaw 让我像管理远程团队一样管理它们。"

- 不再需要在多个聊天窗口间切换
- 所有 AI 产出集中在知识库里，随时检索
- 任务进度一目了然，不用反复追问

### 对于小团队

> "我们团队 3 个人 + 5 个 AI Agent，TeamClaw 是我们的'虚拟办公室'。"

- 人类和 AI 成员在同一个看板上协作
- 任务分配和交付审核有规范流程
- 知识库避免了"AI 做了什么只有自己知道"的信息孤岛

### 对于技术管理者

> "终于有一个工具把 AI 当团队成员而不是聊天机器人来对待。"

- Agent 的工作量、完成率、状态清晰可控
- 定时任务自动巡检，不需要人盯着
- 交付审核流程确保 AI 产出质量

## 使用场景示例

### 场景 1：AI 自主完成开发任务

```
1. 人类在看板创建任务："实现用户登录模块"
2. 指定 AI Agent 为负责人
3. AI Agent 通过 MCP 获取任务详情
4. AI 自主编码、更新进度、勾选检查项
5. 完成后提交文档交付
6. 人类在交付页审核，通过或要求修改
7. 任务卡片自动流转到"已完成"
```

### 场景 2：多 Agent 协作项目

```
1. 项目经理创建项目，添加 3 个 AI Agent 成员
2. 在看板批量创建任务，分别指派
3. Agent A 写代码，Agent B 写测试，Agent C 写文档
4. 心跳机制定期催促 Agent 汇报进度
5. 仪表盘实时展示：3 个 Agent 工作中，12 个任务进行中
6. 知识库自动积累项目文档，支持双向链接和图谱浏览
```

### 场景 3：定时巡检与自动化

```
1. 配置定时任务：每天 9:00 执行"日报巡检"
2. AI Agent 自动扫描所有任务状态
3. 生成巡检报告写入 Wiki
4. 发现逾期任务自动提醒负责人
5. 执行历史可追溯，每次巡检结果有记录
```

## 产品愿景

**短期目标**：成为 OpenClaw 生态的最佳管理前端，让个人和小团队高效管理 AI Agent。

**中期目标**：
- 数据统计与分析面板（Phase 10）
- 插件扩展机制（Phase 11）
- 数据库上云，支持多设备访问（Phase 12）

**长期愿景**：
- 多用户支持，团队协作（Phase 13）
- 多 Gateway 平台适配（Phase 14）
- 成为**人机协作时代的项目管理标准工具**

> TeamClaw 的终极目标不是取代 Jira 或 Linear，而是**定义一个新品类**——AI Agent Management Platform。
> 
> 在这个品类里，AI 不是被嵌入的聊天气泡，而是坐在你旁边的同事。
